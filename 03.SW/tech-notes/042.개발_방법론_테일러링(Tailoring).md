# 1. 개발 방법론 테일러링(Tailoring)의 개요 및 배경

### 가. 개발 방법론 테일러링의 정의
- 프로젝트의 특성(규모, 위험도, 난이도 등)에 맞추어 표준 소프트웨어 개발 방법론의 절차, 기법, 산출물을 **수정, 보완, 가공**하여 최적화된 방법론을 도출하는 활동임
- "One Size Fits All"의 한계를 극복하고, 프로젝트 수행의 **효율성**과 **품질**을 동시에 확보하기 위한 방법론 가공 기법임

### 나. 등장 배경 및 필요성
- **방법론의 유연성 확보**: 표준 방법론은 범용적으로 설계되어 특정 프로젝트에 그대로 적용하기에는 비효율적임
- **자원 최적화**: 불필요한 공정이나 산출물을 제거하여 비용과 시간을 절감하고 핵심 작업에 집중 필요
- **리스크 관리**: 프로젝트의 기술적/관리적 특성에 따라 검증 및 승인 절차를 강화하거나 완화하여 위험 대응

# 2. 테일러링의 프로세스 및 핵심 구성 요소

### 가. 개발 방법론 테일러링 수행 절차 개념도
```mermaid
graph TD
    Standard[표준 방법론 선정] --> Analyze[프로젝트 환경 분석<br/>내부/외부 요인]
    Analyze --> Base[기본 테일러링 가이드라인 적용]
    Base --> Apply[수정/추가/삭제/간소화 수행]
    Apply --> Review[전문가 검토 및 승인]
    Review --> Execute[테일러링된 방법론 확정/적용]

    subgraph "Tailoring Rule"
        Add[추가: 특수 공정 삽입]
        Delete[삭제: 불필요 산출물 제거]
        Modify[수정: 절차 및 양식 변경]
    end
    
    Apply -.-> Tailoring Rule
```

### 나. 테일러링 고려 대상 요인 (Selection Criteria)
| 구분 | 고려 요인 | 상세 내용 |
| :--- | :--- | :--- |
| **내적 요인** | **목표 및 환경** | 프로젝트의 비즈니스 목표, 시스템 유형(Web, 모바일, 패키지 등) |
| | **구성원 역량** | 참여 인력의 기술 숙련도, 방법론에 대한 이해도 |
| | **프로젝트 특성** | 사업 규모(MM), 예산, 기간, 위험도, 중요도 |
| **외적 요인** | **법적 규제** | 컴플라이언스(보안, 개인정보보호법), 표준 지침(전자정부 가이드 등) |
| | **국제 표준** | ISO/IEC 12207, CMMI, SPICE 등 인증 요건 |
| | **인프라 환경** | 개발 도구(CASE), 서버 및 네트워크 인프라, 협업 툴 |

# 3. 테일러링의 핵심 기법 및 비교 분석

### 가. 주요 테일러링 기법
1) **삭제 (Deletion)**: 프로젝트에 해당하지 않는 활동이나 중복된 산출물을 식별하여 제거
2) **추가 (Addition)**: 신기술 도입이나 특정 보안 요건 충족을 위해 표준에 없는 공정 삽입
3) **수정 (Modification)**: 작업의 순서를 조정하거나, 승인 절차(Gate)를 강화/완화하여 변경
4) **병합 (Merging)**: 소규모 프로젝트에서 유사한 공정이나 산출물을 하나로 통합

### 나. 표준 방법론 vs 테일러링된 방법론 비교
| 비교 항목 | 표준 방법론 (Standard) | 테일러링된 방법론 (Tailored) |
| :--- | :--- | :--- |
| **범위** | 전사적, 범용적 | **특정 프로젝트 전용** |
| **목적** | 일관성 및 베스트 프랙티스 제공 | **생산성 및 품질 최적화** |
| **강제성** | 원칙 준수 의무 | 자율적 가공 인정 |
| **효율성** | 낮음 (불필요 공정 포함 가능) | **높음 (핵심 공정 집중)** |
| **적용 시점** | 프로젝트 초기 | 환경 분석 후 실행 단계 전 |

# 4. 기술사적 제언 및 실무 적용 방안

### 가. 실무 도입 시 고려사항: 거버넌스 준수
- **문서화(Tailoring Log)**: 어떤 근거로 산출물을 삭제했는지 기록을 남겨 향후 품질 감사(Audit)나 감리에 대응해야 함
- **적정성 검토**: 무분별한 간소화는 품질 저하를 야기하므로, 품질보증(QA) 담당자나 감리인의 기술적 검토 필수

### 나. 보안(Security) 및 규제 통제 방안
- **보안 가이드라인 유지**: 생산성을 위해 보안 관련 공정(취약점 진단, 암호화 설계 등)을 임의로 삭제하는 행위 엄격히 제한
- **컴플라이언스 매핑**: 테일러링 후에도 산업별 규제(예: 금융보안 가이드)를 만족하는지 추적성(Traceability) 검증

### 다. 최신 트렌드와의 연계
- **Agile 테일러링**: 고정된 스크럼(Scrum) 프레임워크를 조직의 상황에 맞게 변형하는 **Scrum-ban**이나 하이브리드 모델로의 확장 활발
- **Tool-chain 연계**: Jira나 Confluence와 같은 협업 도구와 연동하여 테일러링된 워크플로우를 자동화된 파이프라인으로 구현

> [!tip] **기술사 인사이트**
> 테일러링의 정수는 **"최소의 투입으로 최대의 품질을 뽑아내는 최적화"**입니다. 기술사 답안에서는 테일러링이 단순한 '일 줄이기'가 아니라, 프로젝트의 **리스크(Risk)**를 분석하고 이에 가장 적합한 **무기(방법론)**를 다듬는 전략적 관리 프로세스임을 강조해야 합니다.
