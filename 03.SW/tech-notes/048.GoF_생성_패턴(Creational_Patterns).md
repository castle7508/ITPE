# 1. GoF 생성 패턴(Creational Patterns)의 개요

### 가. 생성 패턴의 정의
- 객체의 생성 프로세스를 추상화하여, 객체가 생성되거나 구성되는 방법으로부터 시스템을 독립시키는 디자인 패턴임
- 객체 생성의 책임을 별도의 클래스나 메서드로 분리함으로써 시스템의 유연성과 재사용성을 높이는 데 목적이 있음

### 나. 주요 목적 및 효과
- **캡슐화**: 구체적인 클래스 정보를 숨기고 인터페이스를 통해 객체 생성
- **결합도 저하**: 클라이언트 코드가 특정 클래스의 인스턴스화 과정에 의존하지 않도록 분리
- **복잡성 관리**: 복합 객체의 생성 과정을 단계별로 나누어 관리 가능

# 2. 생성 패턴의 핵심 5가지 상세 분석

| 패턴명 | 핵심 정의 및 특징 | 주요 구성 요소 |
| :--- | :--- | :--- |
| **Singleton** | 클래스 인스턴스를 단 하나만 생성하고 전역 접근 제공 | static instance, getInstance() |
| **Factory Method** | 객체 생성의 결정을 서브클래스에 위임하여 유연성 극대화 | Creator, Product, ConcreteCreator |
| **Abstract Factory** | 연관된 객체들의 집합을 생성하는 추상 인터페이스 제공 | AbstractFactory, AbstractProduct |
| **Builder** | 복합 객체의 생성 과정과 표현을 분리하여 동일 공정으로 다른 표현 생성 | Builder, Director, ConcreteBuilder |
| **Prototype** | 원본 객체를 복사(Clone)하여 새로운 객체를 생성하는 방식 | Prototype, ConcretePrototype, Clone() |

# 3. 상세 기술 및 패턴별 심화 분석

### 가. Factory Method vs Abstract Factory
- **Factory Method**: 단일 객체의 생성에 집중하며, 상속을 통해 하위 클래스에서 구체적인 타입을 결정함
- **Abstract Factory**: 연관된 여러 객체군(Family)의 생성에 집중하며, 합성을 통해 관련 객체들을 한꺼번에 교체하기 용이함

### 나. Builder vs Prototype
- **Builder**: 복잡한 인스턴스를 조립하여 만드는 과정에 초점 (Step-by-Step 생성)
- **Prototype**: 생성 비용이 크거나 복잡한 객체를 기존 인스턴스 복제를 통해 빠르게 생성하는 데 초점

# 4. 기술사적 제언 및 실무 적용 방안

### 가. 실무 도입 시 고려사항
- **Singleton의 주의점**: 멀티스레드 환경에서 Thread-safe하게 구현(Double-checked locking 등)해야 하며, 과도한 사용 시 전역 상태로 인한 테스트 어려움 발생
- **Builder 패턴 활용**: 매개변수가 많은 생성자의 경우 가독성과 안정성을 위해 필수적으로 사용 권장 (Effective Java 스타일)

### 나. 보안 및 품질 관리
- **객체 생성 통제**: 인가되지 않은 객체 생성을 방지하기 위해 생성자 접근 제어자를 활용하고, 오직 정의된 팩토리를 통해서만 인스턴스를 획득하도록 강제
- **리소스 최적화**: Flyweight 패턴과 연계하여 생성 패턴 내에서 객체 풀링(Pooling)을 구현함으로써 메모리 낭비 방지

> [!tip] **기술사 인사이트**
> 생성 패턴의 본질은 **"누가, 언제, 무엇을 만드는가"**에 대한 통제권을 획득하는 것입니다. 특히 **Abstract Factory**와 **Builder**는 시스템의 확장성과 설정 유연성을 결정짓는 핵심 패턴으로, 대규모 프레임워크 설계 시 반드시 고려되어야 할 핵심 기술입니다.
