# 1. 상속성(Inheritance)의 개요 및 배경

### 가. 상속성의 정의
- 상위 클래스(부모)의 모든 속성과 기능을 하위 클래스(자식)가 물려받아 그대로 사용하거나, 필요에 따라 기능을 확장 또는 재정의하여 사용할 수 있는 객체지향의 핵심 원리임
- 기존 클래스를 기반으로 새로운 클래스를 생성함으로써 코드의 **재사용성(Reusability)**을 높이고, 객체 간의 **계층 구조(Hierarchy)**를 형성하는 기술임

### 나. 등장 배경 및 필요성
- **코드 중복 제거**: 여러 클래스에 공통적으로 존재하는 속성과 기능을 상위 클래스로 통합하여 관리 효율성 증대
- **소프트웨어 생산성 향상**: 검증된 기존 코드를 수정 없이 활용함으로써 개발 기간 단축 및 오류 발생 가능성 차단
- **분류 체계의 정립**: 현실 세계의 복잡한 개체들을 **일반화(Generalization)**와 **특수화(Specialization)**의 논리적 관계로 구조화

# 2. 상속성의 아키텍처 및 핵심 메커니즘

### 나. 핵심 구성 요소 및 유형
| 구분 | 명칭 | 상세 설명 |
| :--- | :--- | :--- |
| **구성 요소** | **Super Class** | 상속을 해주는 부모 클래스, 공통적인 특징을 가짐 (Base Class) |
| | **Sub Class** | 상속을 받는 자식 클래스, 구체적인 특징을 추가함 (Derived Class) |
| **상속 유형** | **단일 상속** | 하나의 클래스가 오직 하나의 부모로부터 상속받음 (Java 등) |
| | **다중 상속** | 두 개 이상의 부모로부터 상속받음 (C++ 등, 다이아몬드 문제 발생 가능) |
| | **인터페이스 상속** | 행위의 규격(Interface)만을 상속받아 구현을 강제함 |

# 3. 상세 기술 및 상속 vs 합성 비교 분석

### 가. 상속의 오용과 취약한 기반 클래스 문제(Fragile Base Class)
- **강결합(Tight Coupling)**: 부모 클래스의 변경이 자식 클래스에게 예기치 못한 영향을 주어 시스템 안정성을 해칠 수 있음
- **불필요한 기능 노출**: 자식 클래스에게 적합하지 않은 부모의 공개 메서드가 상속되어 캡슐화가 깨질 수 있음

### 나. 상속(Inheritance) vs 합성(Composition) 비교 분석
| 비교 항목 | 상속 (Inheritance) | 합성 (Composition) |
| :--- | :--- | :--- |
| **관계 유형** | **IS-A 관계** (이다) | **HAS-A 관계** (가진다) |
| **바인딩** | 정적 바인딩 (Compile-time) | **동적 바인딩** (Run-time) |
| **결합도** | **높음** (부모-자식 간 강력한 의존) | **낮음** (객체 주입을 통한 느슨한 연결) |
| **캡슐화** | 화이트박스 재사용 (내부 노출) | **블랙박스 재사용** (인터페이스 소통) |
| **유연성** | 낮음 (런타임에 부모 변경 불가) | **높음** (실행 중 동적 교체 가능) |

# 4. 기술사적 제언 및 실무 적용 방안

### 가. 실무 도입 시 고려사항: LSP(리스코프 치환 원칙) 준수
- 상속은 반드시 **논리적인 IS-A 관계**가 성립할 때만 사용해야 하며, 단순한 코드 재사용을 목적으로 상속을 남발해서는 안 됨
- 자식 객체는 언제나 부모 객체로 치환되어도 프로그램의 의도대로 동작해야 함을 설계 단계에서 검증 필수

### 나. 거버넌스 및 설계 통제 방안
- **Composition over Inheritance**: 상속보다는 합성을 우선적으로 고려하여 시스템의 유연성을 확보하고, 상속 계층은 최대 3~4단계 이내로 제한
- **상속 금지 제어**: 상속을 의도하지 않은 클래스는 `final` 키워드(Java) 등을 사용하여 무분별한 확장을 방지하는 보안 코딩 정책 수립

### 다. 현대적 발전 방향
- **Mixins 및 Traits**: 현대적 언어(Go, Rust, Scala 등)에서는 다중 상속의 폐해를 막기 위해 기능을 조립하는 형식의 **Mixin**이나 **Trait** 구조를 선호하는 추세
- **클린 아키텍처 연계**: 도메인 엔티티 설계 시 상속보다는 **인터페이스 구현**을 통해 외부 기술과의 의존성을 분리하는 것이 정석임

> [!tip] **기술사 인사이트**
> 상속은 소프트웨어의 **"분류와 재사용"**을 위한 강력한 도구이지만, 양날의 검과 같습니다. 기술사 답안에서는 상속의 편리함보다 상속으로 인한 **"변경의 전파"** 문제를 인식하고, 이를 극복하기 위한 **합성(Composition)** 및 **인터페이스** 기반 설계 능력을 보여주는 것이 핵심 차별화 포인트입니다.
