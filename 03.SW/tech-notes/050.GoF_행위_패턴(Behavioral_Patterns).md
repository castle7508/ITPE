# 1. GoF 행위 패턴(Behavioral Patterns)의 개요

### 가. 행위 패턴의 정의
- 객체나 클래스 간의 상호작용 방법과 책임 분담을 정의하는 패턴으로, 실행 시점에 동적으로 변하는 객체 간의 통신 복잡성을 해결하는 데 목적이 있음
- 알고리즘뿐만 아니라 객체 간의 **협력 관계(Collaboration)**를 정형화하여 유연하고 확장 가능한 설계를 지원함

### 나. 주요 목적 및 효과
- **결합도 분리**: 메시지 송신자와 수신자 간의 의존성 제거
- **알고리즘 캡슐화**: 상황에 따라 알고리즘을 선택하거나 교체 가능
- **상태 관리 자동화**: 객체의 상태 변화에 따른 복잡한 로직 체계화

# 2. 행위 패턴의 주요 11가지 상세 분석

| 패턴명 | 핵심 정의 및 특징 | 주요 구성 요소 |
| :--- | :--- | :--- |
| **Strategy** | 상황에 따라 알고리즘을 인터페이스 기반으로 동적 교체 | Context, Strategy |
| **Observer** | 객체 상태 변경 시 구독 중인 다수 객체에 자동 통지 | Subject, Observer |
| **Template Method** | 상위에서 전체 흐름 규정, 하위에서 세부 단계 구현 | AbstractClass, ConcreteClass |
| **State** | 객체의 상태를 클래스화하여 상태에 따른 행위 분리 | Context, State |
| **Command** | 요청을 객체로 캡슐화하여 큐잉, 로깅, 취소(Undo) 지원 | Command, Invoker, Receiver |
| **Iterator** | 내부 구조 노출 없이 집합체 원소에 순차적 접근 제공 | Aggregate, Iterator |
| **Mediator** | 객체 간의 복잡한 통신을 하나의 중개자 객체에 위임 | Mediator, Colleague |
| **Memento** | 캡슐화를 유지하며 객체 상태를 저장하여 나중에 복구 | Originator, Memento |
| **Visitor** | 데이터 구조와 행위(기능 처리)를 분리하여 확장성 확보 | Visitor, Element |
| **Interpreter** | 특정 언어의 문법을 정의하고 해석하는 방법 제시 | Expression, Context |
| **CoR** | 한 요청을 여러 객체가 연속적으로 처리 (책임 떠넘기기) | Handler, Client |

# 3. 상세 기술 및 빈출 패턴 심화 분석

### 가. Strategy vs State
- **Strategy**: 클라이언트가 **알고리즘을 선택**하여 Context에 주입함 (수동 교체)
- **State**: 객체 내부의 **상태 변화에 따라** 스스로 행위 클래스를 전환함 (자동 교체)

### 나. Observer 패턴의 전파 방식 (Push vs Pull)
- **Push 모델**: Subject가 변경된 데이터를 Observer에게 **강제로 밀어넣음** (실시간성 우수)
- **Pull 모델**: 변경 통지만 받은 후 Observer가 필요한 데이터를 **직접 가져감** (유연성 우수)

# 4. 기술사적 제언 및 실무 적용 방안

### 가. 실무 도입 시 고려사항
- **Template Method의 활용**: 프레임워크의 라이프사이클(Init, Render, Destroy) 설계 시 가장 기본이 되는 패턴임
- **Undo/Redo 구현**: 복잡한 시스템의 취소 기능을 위해 **Command** 패턴(명령 저장)과 **Memento** 패턴(상태 저장)을 조합하여 설계

### 나. 거버넌스 및 설계 통제 방안
- **LSP(리스코프 치환) 준수**: 행위 패턴 적용 시 하위 타입이 상위 계약을 깨뜨리지 않도록 철저한 단위 테스트 및 검증 필요
- **메시지 순서 보장**: 분산 환경에서 Observer나 Command 패턴 적용 시 메시지 큐와 연계하여 **멱등성(Idempotency)** 및 순서(Ordering) 보장 대책 수립

> [!tip] **기술사 인사이트**
> 행위 패턴은 **"유연한 메시징"**의 핵심입니다. 현대 프로그래밍 언어의 람다(Lambda)와 함수형 인터페이스는 기존의 복잡한 행위 패턴들을 매우 간결한 코드로 대체하고 있으므로, 패턴의 고전적 구조와 현대적 변천 과정을 동시에 기술하는 것이 차별화 포인트입니다.
