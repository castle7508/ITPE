# 소프트웨어 테스트 유형

> [!info] **소프트웨어 테스트 유형이란?**
> 테스트의 목적, 대상, 시기, 방법 등에 따라 테스트 활동을 체계적으로 분류한 것입니다. 적절한 테스트 유형을 선택하고 조합하는 것은 한정된 자원으로 결함 발견율을 극대화하는 핵심 전략입니다.

---

## 1. 소프트웨어 테스트 유형의 개요
### 가. 테스트 분류의 필요성
- 시스템의 복잡도가 증가함에 따라 기능뿐만 아니라 성능, 보안, 가용성 등 다양한 관점의 검증이 필요함

### 나. 분류 기준 (Standard)
1. **수행 단계**: 개발 주기(V-모델)에 따른 분류
2. **테스트 대상**: 프로그램 내부 구조 노출 여부에 따른 분류
3. **테스트 목적**: 확인하고자 하는 품질 특성에 따른 분류
4. **실행 방식**: 프로그램 실행 여부에 따른 분류

---

## 2. 주요 테스트 유형 상세 (What & How)
### 가. 프로그램 구조 노출 기준 (Black-box vs White-box)

| 구분 | 화이트박스 테스트 (White-box) | 블랙박스 테스트 (Black-box) |
| :--- | :--- | :--- |
| **대상** | 소스 코드 내부 로직, 제어 흐름 | 요구사항 명세, 인터페이스 |
| **관점** | 개발자 관점 | 사용자 관점 |
| **기법** | 구문/조건/결정 커버리지 등 | 경계값 분석, 동등 분할, 결정 테이블 등 |
| **단계** | 주로 단위 테스트 단계 | 시스템 테스트, 인수 테스트 단계 |

### 나. 테스트 목적에 따른 분류 (Behavioral)

| 유형 | 설명 | 핵심 포인트 |
| :--- | :--- | :--- |
| **회귀 테스트 (Regression)** | 수정된 코드에 의한 의도치 않은 결함 발생 여부 확인 | **자동화**가 필수적임 |
| **부하/성능 테스트** | 시스템의 처리량, 응답시간, 임계치 확인 | JMeter, nGrinder 활용 |
| **보안 테스트** | 불법적 침입 및 권한 오용 가능성 점검 | SQL Injection, XSS 등 |
| **스모크 테스트** | 주요 기능의 작동 여부를 확인하는 기초 테스트 | Build 확인용 (산소 검사) |

---

## 3. 실행 방식 및 정적/동적 테스트
### 가. 테스트 수행 방식 (Mermaid)

```mermaid
graph TD
    A[SW Testing Methods] --> B[Static Testing]
    A --> C[Dynamic Testing]

    B --> B1[Review / Inspection / Walkthrough]
    B --> B2[Static Analysis Tools]
    
    C --> C1[Structure-based (White-box)]
    C --> C2[Specification-based (Black-box)]
    C --> C3[Experience-based (Exploratory)]
```

### 나. 정적 테스트의 중요성
- 소스 코드를 실행하지 않고 명세서나 코드를 검토하여 결함을 조기에 발견함
- 전체 결함의 약 30~50%를 정적 테스트(인스펙션 등)를 통해 제거 가능함

---

## 4. 기술사적 제언 및 실무 적용 방안
### 가. 테스트 유형의 전략적 조합 (Test Mix)
- 모든 유형을 테스트하는 것은 불가능하므로, 리스크 분석 결과를 토대로 **Test Pyramid** 모델을 참고하여 단위 테스트는 많이, 사용자 테스트는 핵심 위주로 배치해야 함

### 나. 기술사적 인사이트
- **Regression Testing Strategy**: 변경 영향 분석을 통해 수정된 부분뿐만 아니라 연관된 핵심 기능을 선별하여 효율적인 회귀 테스트를 수행해야 함
- **Experience-based Testing**: 명세 기반 테스트의 한계를 극복하기 위해 테스터의 도메인 지식을 활용한 **탐색적 테스팅**을 적절히 혼합하여 변칙적인 결함을 적발해야 함
