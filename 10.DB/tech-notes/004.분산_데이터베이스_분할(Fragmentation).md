# 1. 분산 데이터베이스 분할 (Fragmentation)의 개요

## 가. 정의
- 하나의 논리적 데이터베이스를 물리적으로 여러 개의 단위로 나누어 서로 다른 사이트(Node)에 분산 저장하는 기법
- 분산 데이터베이스의 투명성(Transparency) 중 **분할 투명성**을 보장하기 위한 핵심 메커니즘

## 나. 분할의 필요성 및 목적
- **지역적 자치성**: 각 지역(Node)에서 자주 사용하는 데이터를 로컬에 위치시켜 처리 효율성 증대
- **부하 분산 (Load Balancing)**: 특정 서버에 집중되는 트래픽을 분산하여 전체 시스템 성능 향상
- **가용성 증대**: 특정 노드 장애 시에도 다른 노드의 데이터를 통해 서비스 지속성 확보
- **병렬 처리**: 분할된 데이터를 여러 노드에서 동시에 처리하여 응답 시간 단축

# 2. 데이터베이스 분할 단위 및 주요 방식

## 가. 데이터베이스 분할 단위
분산 데이터베이스에서 데이터를 나누는 기본 단위를 의미하며, 주로 **릴레이션(Relation)** 또는 **속성(Attribute)** 집합을 기준으로 합니다.

| 분할 단위 | 설명 | 특징 |
|---|---|---|
| **릴레이션 (Relation)** | 테이블 전체를 하나의 단위로 분할하여 배치 | 가장 일반적인 단위 |
| **튜플 (Tuple)** | 행(Row)을 기준으로 데이터를 나누어 여러 사이트에 분산 | 수평 분할의 기본 단위 |
| **속성 (Attribute)** | 열(Column)을 기준으로 성격이 다른 속성끼리 묶어 분산 | 수직 분할의 기본 단위 |
| **복합 단위 (Hybrid)** | 수평과 수직 분할을 혼합하여 더 작은 조각(Fragment) 생성 | 복잡한 쿼리 최적화에 유리 |

# 3. 수평 분할과 수직 분할의 상세 비교

분할 방식은 릴레이션을 쪼개는 방향에 따라 크게 두 가지로 구분됩니다. **[두음: 수수혼]**

## 가. 수평 분할 (Horizontal Fragmentation)
- **개념**: 릴레이션의 **행(Row, Tuple)**을 기준으로 분할하여 서로 다른 노드에 저장
- **특징**: 모든 분할된 조각(Fragment)은 동일한 스키마(Schema)를 공유함
- **매커니즘**: `SELECT` 연산과 유사하며, 특정 조건(예: 지역, 날짜)에 따라 분리
- **장점**: 병렬 처리에 유리하며, 특정 조건에 부합하는 데이터 접근 속도가 빠름

## 나. 수직 분할 (Vertical Fragmentation)
- **개념**: 릴레이션의 **열(Column, Attribute)**을 기준으로 분할하여 서로 다른 노드에 저장
- **특징**: 분할된 각 조각은 서로 다른 스키마를 가지며, **기본키(Primary Key)**를 포함해야 함 (재구성 시 Join 필요)
- **매커니즘**: `PROJECT` 연산과 유사하며, 업무 성격이나 보안 등급에 따라 분리
- **장점**: 불필요한 속성 조회를 줄여 I/O 효율성을 높이고, 특정 속성에 대한 보안 통제 용이

## 다. 상세 비교표
| 비교 항목 | 수평 분할 (Horizontal) | 수직 분할 (Vertical) |
|---|---|---|
| **분할 기준** | 튜플 (행, Row) | 속성 (열, Column) |
| **연산 방식** | Selection ($\sigma$) | Projection ($\pi$) |
| **데이터 스키마** | 모든 조각의 스키마가 동일 | 조각마다 스키마가 다름 |
| **데이터 재구성** | Union ($\cup$) 연산 활용 | Join ($\bowtie$) 연산 활용 |
| **무결성 조건** | 완전성, 재구성, 상호 배제성 | **기본키(PK) 중복 포함 필수** |
| **주요 활용** | 샤딩(Sharding), 지역별 데이터 분산 | 테이블 정규화, 칼럼별 보안 강화 |

# 4. 분할 시 준수해야 할 3대 원칙 (Correctness Rules)

분할된 데이터는 논리적으로 원래의 릴레이션과 동일함을 보장해야 합니다. **[두음: 완재배]**

1.  **완전성 (Completeness)**: 전체 릴레이션 $R$의 모든 데이터는 반드시 최소 하나 이상의 조각 $R_i$에 포함되어야 함
2.  **재구성 (Reconstructability)**: 분할된 조각들을 적절한 관계 연산(Union, Join)을 통해 원래의 릴레이션으로 복구할 수 있어야 함
3.  **상호 배제성 (Disjointness)**: (수평 분할 시) 하나의 튜플은 오직 하나의 조각에만 속해야 함 (중복 방지)

> [!tip] **기술사 인사이트**
> 현대의 클라우드 기반 대규모 시스템에서는 수평 분할인 **샤딩(Sharding)**이 성능 확장의 핵심 기술로 자리 잡았습니다. 기술사는 분산 DB 설계 시 **데이터 국부성(Locality)**과 **네트워크 오버헤드** 사이의 트레이드-오프(Trade-off)를 정밀하게 분석하여 최적의 분할 전략을 제시할 수 있어야 합니다.
