# 1. GoF 구조 패턴(Structural Patterns)의 개요

### 가. 구조 패턴의 정의
- 클래스나 객체를 조합하여 더 큰 구조를 형성하는 패턴으로, 서로 다른 인터페이스를 가진 객체들이 협력할 수 있도록 하거나 복잡한 시스템을 단순화하는 기법임
- 상속(Inheritance)보다는 **합성(Composition)**을 활용하여 유연한 객체 구조를 설계하는 데 목적이 있음

### 나. 주요 목적 및 효과
- **상호운용성 확보**: 호환되지 않는 인터페이스 간의 연결 지원
- **추상화 및 단순화**: 복잡한 하위 시스템에 대한 단순한 인터페이스 제공
- **동적 기능 확장**: 객체의 결합을 통해 런타임에 새로운 기능 추가

# 2. 구조 패턴의 핵심 7가지 상세 분석

| 패턴명 | 핵심 정의 및 특징 | 주요 구성 요소 |
| :--- | :--- | :--- |
| **Adapter** | 호환되지 않는 인터페이스를 중간에서 변환하여 재활용 | Target, Adaptee, Adapter |
| **Bridge** | 구현부에서 추상층을 분리하여 각자 독립적으로 확장 가능 | Abstraction, Implementor |
| **Composite** | 개별 객체와 복합 객체를 동일하게 다루도록 트리 구조 형성 | Component, Leaf, Composite |
| **Decorator** | 객체에 동적으로 새로운 책임(기능)을 추가하는 방식 | Component, Decorator |
| **Facade** | 서브 시스템의 복잡한 인터페이스들을 하나의 단순한 창구로 통합 | Facade, Subsystem Classes |
| **Flyweight** | 다수의 작은 객체들을 공유하여 메모리 사용량 최적화 | FlyweightFactory, Flyweight |
| **Proxy** | 실제 객체에 대한 접근을 제어하거나 대리자를 통해 기능 수행 | Subject, RealSubject, Proxy |

# 3. 상세 기술 및 유사 패턴 비교 분석

### 가. Proxy vs Adapter vs Facade
- **Proxy**: 인터페이스를 변경하지 않고 **접근 제어**나 **부가 기능**을 추가함
- **Adapter**: 인터페이스를 **다른 형태로 변환**하여 기존 코드를 재사용함
- **Facade**: 여러 인터페이스를 **단순화**하여 사용하기 편하게 만듦

### 나. Decorator vs Bridge
- **Decorator**: 객체의 **외보 기능 확장**에 초점을 맞추며 재귀적 합성을 활용함
- **Bridge**: 기능의 계층과 구현의 계층을 **완전히 분리**하여 독립적인 변형을 허용함

# 4. 기술사적 제언 및 실무 적용 방안

### 가. 실무 도입 시 고려사항
- **Composite 패턴**: 파일 시스템이나 UI 컴포넌트 트리와 같이 **재귀적 구조**를 가진 도메인에 필수적으로 적용
- **Proxy 패턴 활용**: AOP(관점 지향 프로그래밍)의 핵심 기술로, 로깅, 트랜잭션, 지연 로딩(Virtual Proxy) 구현 시 적극 활용

### 나. 거버넌스 및 통제 방안
- **인터페이스 일관성**: 구조 패턴 적용 시 클라이언트가 구체 클래스에 의존하지 않도록 인터페이스(Target/Subject) 기반 통신 강제
- **리소스 모니터링**: Flyweight 패턴 적용 시 공유 객체 풀(Pool)의 크기와 생명주기를 관리하여 메모리 누수 방지

> [!tip] **기술사 인사이트**
> 구조 패턴의 정수는 **"인터페이스의 유연한 활용"**입니다. 특히 **Facade**는 마이크로서비스(MSA)의 **API Gateway** 사상으로 연결되며, **Proxy**는 분산 컴퓨팅의 투명성(Transparency)을 확보하는 핵심 기법임을 답안에서 강조해야 합니다.
