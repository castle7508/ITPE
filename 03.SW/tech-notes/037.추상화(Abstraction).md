# 1. 추상화(Abstraction)의 개요 및 배경

### 가. 추상화의 정의
- 복잡한 현실 세계의 실체(Entity)에서 **불필요한 세부 사항은 제거**하고, 해결하고자 하는 문제 영역에서 가장 **본질적인 특징(속성 및 행위)만을 추출**하여 모델링하는 객체지향의 핵심 원리임
- "무엇(What)"을 수행하는지에 집중하고 "어떻게(How)" 구현하는지는 하위 계층이나 구체적인 객체로 위임하는 **관심사의 분리(SoC)** 기술임

### 나. 등장 배경 및 필요성
- **복잡성 관리(Complexity Management)**: 시스템의 규모가 커짐에 따라 전체를 한눈에 파악하기 위해 상세 구현을 가리고 상위 수준의 개념으로 일반화할 필요성 증대
- **유연한 구조 확보**: 구체적인 구현체(Detail)가 아닌 추상적인 개념(Policy)에 의존하게 함으로써 시스템의 변경에 유연하게 대응함
- **재사용성 및 표준화**: 공통된 특징을 상위 계층으로 추출하여 코드 중복을 제거하고 표준 인터페이스를 정립함

# 2. 추상화의 아키텍처 및 핵심 메커니즘

### 나. 추상화의 3대 유형
| 유형 | 상세 내용 | 예시 |
| :--- | :--- | :--- |
| **데이터 추상화** | 객체의 속성과 상태를 핵심 데이터 위주로 모델링 | 사원 객체의 (이름, 사번, 급여) |
| **과정(기능) 추상화** | 복잡한 로직을 하나의 이름(메서드) 뒤에 숨김 | `calculateTax()`, `startEngine()` |
| **제어 추상화** | 실행 흐름의 세부 사항을 감추고 논리적 흐름만 표현 | Loop, Exception Handling, Frameworks |

# 3. 상세 기술 및 추상 클래스 vs 인터페이스 비교

### 가. 상세 동작 메커니즘: 일반화(Generalization)와 특수화(Specialization)
1) **일반화 (Top-down)**: 여러 하위 클래스의 공통점을 찾아 상위 클래스로 묶는 과정 (추상화의 과정)
2) **특수화 (Bottom-up)**: 상위 클래스를 상속받아 구체적인 구현과 고유 특성을 추가하는 과정 (구체화의 과정)

### 나. 추상 클래스(Abstract Class) vs 인터페이스(Interface) 비교 분석
| 비교 항목 | 추상 클래스 (Abstract Class) | 인터페이스 (Interface) |
| :--- | :--- | :--- |
| **핵심 목적** | **IS-A 관계**, 관련성 높은 클래스 간 상속 | **CAN-DO 관계**, 서로 다른 클래스 간 행위 표준 |
| **구현 포함 여부** | 일반 메서드 + 추상 메서드 포함 가능 | 오직 추상 메서드만 가능 (Default 제외) |
| **다중 상속** | 불가능 (단일 상속만 허용) | **가능 (다중 구현 허용)** |
| **변수 특성** | 일반 변수(필드) 선언 가능 | 상수(`public static final`)만 가능 |
| **의미적 관점** | **확장(Extend)** 및 상태 공유 | **규격(Specification)** 및 계약 준수 |

# 4. 기술사적 제언 및 실무 적용 방안

### 가. 실무 도입 시 고려사항: 적정 추상화 수준(Granularity)
- **추상화의 함정**: 지나친 추상화는 오히려 코드 읽기를 어렵게 만들고 추적성을 저하시키므로, 비즈니스 가치가 높은 영역에 집중하여 적용해야 함
- **YAGNI (You Ain't Gonna Need It)**: 미래의 막연한 변화를 위해 미리 복잡한 추상 계층을 설계하기보다는, 실제 필요성이 발생할 때 리팩토링을 통해 추상화 수행

### 나. 거버넌스 및 보안(Security) 통제 방안
- **인터페이스 기반 보안**: 외부 노출 시 구체 클래스가 아닌 인터페이스(추상화)를 반환하여 내부 구현 정보를 철저히 은닉 (Information Hiding 연계)
- **보안 가이드라인**: 추상 계층에서 입력값 검증(Validation) 정책을 인터페이스 규격으로 정의하여 하위 구현체들이 반드시 보안 규칙을 준수하도록 강제

### 다. 현대적 아키텍처와의 연계
- **DIP (의존성 역전 원칙)**: 추상화는 고수준 정책이 저수준 세부사항에 의존하지 않게 하는 핵심 도구이며, 이는 **클린 아키텍처**와 **MSA** 서비스 간 통신의 표준이 됨
- **폴리모피즘(다형성)의 토대**: 추상화된 부모 타입을 통해 수많은 자식 객체를 일관되게 제어함으로써 시스템의 동적 확장성(Plug-in 아키텍처)을 보장함

> [!tip] **기술사 인사이트**
> 추상화는 **"변하는 것(How)과 변하지 않는 것(What)을 가르는 기준선"**입니다. 기술사 답안에서는 추상화가 단순한 문법적 도구를 넘어, 소프트웨어의 **변경 탄력성(Agility)**과 **테스트 용이성(Testability)**을 확보하기 위한 전략적 경계(Boundary)임을 강조하십시오.
