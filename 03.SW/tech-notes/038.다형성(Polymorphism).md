# 1. 다형성(Polymorphism)의 개요 및 배경

### 가. 다형성의 정의
- 하나의 인터페이스나 메시지가 객체의 실제 타입에 따라 **서로 다른 형태의 결과로 응답**하거나 동작하는 객체지향의 핵심 원리임
- "하나의 이름, 여러 가지 형태(One Name, Many Forms)"를 의미하며, 상속 관계에 있는 클래스들 간의 공통된 행위를 추상화하여 시스템의 유연성을 극대화하는 기법임

### 나. 등장 배경 및 필요성
- **유연성 및 확장성 확보**: 새로운 객체 타입이 추가되더라도 기존의 호출 코드를 수정하지 않고 그대로 재사용하기 위함 (**OCP** 달성)
- **복잡한 조건문 제거**: `if-else`나 `switch-case`를 통한 분기 처리 대신, 객체의 자율적인 행위(Message)를 통해 로직을 수행함으로써 코드 가독성 향상
- **결합도(Coupling) 저하**: 구체적인 클래스 정보가 아닌 상위 타입(Interface/Abstract Class)에 의존하게 하여 모듈 간의 독립성 보장

# 2. 다형성의 아키텍처 및 핵심 메커니즘

### 나. 핵심 구현 기술: 오버로딩 vs 오버라이딩
| 구분 | 오버로딩 (Overloading) | 오버라이딩 (Overriding) |
| :--- | :--- | :--- |
| **핵심 개념** | 한 클래스 내 동일 메서드명 다중 정의 | 상속 관계에서 부모 메서드 재정의 |
| **판별 기준** | 매개변수(개수, 타입)의 차이 | 메서드 시그니처(이름, 파라미터, 반환타입) 동일 |
| **바인딩 시점** | **정적 바인딩** (Compile-time) | **동적 바인딩** (Run-time) |
| **목적** | 사용 편의성, 유사 행위 통합 | **다형성 구현**, 하위 클래스 특수화 |

# 3. 상세 기술 및 바인딩 메커니즘 분석

### 가. 동적 바인딩(Dynamic Binding)과 가상 함수 테이블(vtable)
1) **동적 바인딩**: 컴파일 시점이 아닌, 프로그램 실행 중에 실제 객체의 타입을 확인하여 호출될 함수 주소를 결정하는 방식 (**Late Binding**)
2) **vtable (Virtual Method Table)**: 가상 함수를 포함한 클래스마다 생성되는 메모리 테이블로, 실행 시 실제 구현체의 함수 주소를 찾아가는 인덱스 역할을 수행함 (**컴시응 관점의 로우레벨 메커니즘**)

### 나. 다형성의 유형별 분류
- **유니버설 다형성 (Universal)**:
    - **매개변수 다형성 (Parametric)**: 제네릭(Generic)과 같이 다양한 타입을 하나의 코드로 처리
    - **포함 다형성 (Inclusion)**: 상속 관계를 통한 업캐스팅 및 오버라이딩 활용 (가장 일반적인 다형성)
- **임시 다형성 (Ad-hoc)**:
    - **오버로딩 (Overloading)**: 동일 이름의 함수가 매개변수에 따라 다르게 동작
    - **강제 형변환 (Coercion)**: 암시적/명시적 형변환을 통한 다형적 처리

# 4. 기술사적 제언 및 실무 적용 방안

### 가. 실무 도입 시 고려사항: LSP(리스코프 치환 원칙) 준수
- 자식 클래스는 항상 부모 클래스의 역할을 완벽히 대체할 수 있어야 함
- 부모의 의도와 다른 엉뚱한 행위로 오버라이딩할 경우, 다형성을 활용하는 클라이언트 코드에서 예기치 못한 런타임 오류가 발생할 수 있음에 유의

### 나. 디자인 패턴과의 연계 활용
- **Strategy 패턴**: 알고리즘군을 정의하고 다형성을 통해 필요에 따라 동적으로 교체하여 사용
- **Factory Method 패턴**: 객체 생성 로직을 추상화하여, 클라이언트가 구체 클래스에 의존하지 않고 다형적 객체를 생성받게 함

### 다. 현대적 프로그래밍 트렌드
- **함수형 인터페이스**: 현대의 언어(Java, C# 등)는 단일 추상 메서드(SAM)를 통한 람다식 지원으로 다형성을 함수 단위로 확장하여 표현력을 높임
- **Duck Typing**: Python, Go 등에서는 명시적 상속이 없더라도 인터페이스 규격만 맞으면 다형성을 인정하는 유연한 구조 지향

> [!tip] **기술사 인사이트**
> 다형성은 **"제어 흐름의 추상화"**입니다. 기술사 답안에서는 다형성이 단순히 코드를 예쁘게 만드는 수단이 아니라, 요구사항의 변화가 코드 전반으로 전파되는 것을 차단하는 **"변경의 격리(Isolation)"** 기술임을 강조해야 합니다.
