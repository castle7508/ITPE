# 1. 추상 클래스 및 인터페이스의 개요 및 배경

### 가. 추상 클래스와 인터페이스의 정의
- **추상 클래스(Abstract Class)**: 하나 이상의 추상 메서드를 포함하며 구체적인 클래스들의 공통점을 추출하여 만든, 직접 인스턴스화할 수 없는 **미완성 설계도**임
- **인터페이스(Interface)**: 객체의 행위(Behavior)를 정의하는 규격으로서, 클래스들이 반드시 구현해야 하는 메서드들의 목록을 가진 **표준 계약서**임

### 나. 등장 배경 및 필요성
- **표준화 및 강제성**: 대규모 프로젝트에서 여러 개발자가 협업할 때, 메서드 명칭과 파라미터를 통일하여 시스템의 일관성 확보 필요
- **다형성(Polymorphism) 극대화**: 구체적인 구현이 아닌 타입(Type) 중심의 프로그래밍을 통해 시스템의 유연성과 확장성 확보
- **코드 재사용 및 중복 제거**: 공통 로직을 상위 계층으로 끌어올려 코드 중복을 막고 유지보수 비용 절감

# 2. 추상화 구현의 아키텍처 및 핵심 메커니즘

### 가. 상속(Inheritance)과 구현(Realization)의 구조도

### 나. 핵심 구성 요소 및 특징
| 구분 | 추상 클래스 (Abstract Class) | 인터페이스 (Interface) |
| :--- | :--- | :--- |
| **선언 키워드** | `abstract class` | `interface` |
| **메서드 형태** | 추상 메서드 + 일반 메서드 포함 가능 | 주로 추상 메서드 (Java 8+ default/static 가능) |
| **상속/구현** | `extends` (단일 상속) | `implements` (다중 구현) |
| **변수(필드)** | 일반 멤버 변수(상태) 가질 수 있음 | 오직 상수(`public static final`)만 가능 |
| **핵심 목적** | **확장(Extends)**: 부모의 유전자를 물려받음 | **규격(Specs)**: 약속된 행위를 보장함 |

# 3. 상세 기술 및 심화 비교 분석

### 가. 객체지향 설계 관점의 차이점 심화
1) **추상 클래스 (계층적 추상화)**: 밀접하게 관련된 클래스들 간의 공통 상태와 행위를 공유할 때 사용 (**유연한 IS-A 관계**)
2) **인터페이스 (수평적 추상화)**: 서로 다른 계층의 클래스들이 동일한 기능을 수행해야 할 때 사용 (**엄격한 CAN-DO 계약**)

### 나. 현대적 프로그래밍에서의 변화 (Java 8/9+ 기준)
| 기능 | 인터페이스의 발전 내용 | 효과 |
| :--- | :--- | :--- |
| **Default Method** | 인터페이스 내에 공통 구현 로직 포함 가능 | 하위 클래스 호환성 유지하며 기능 확장 |
| **Static Method** | 헬퍼(Helper) 성격의 유틸리티 메서드 제공 | 인터페이스 단위의 독립적 기능 수행 |
| **Private Method** | 인터페이스 내부에서만 사용하는 로직 캡슐화 | 코드 중복 제거 및 정보 은닉 강화 |

# 4. 기술사적 제언 및 실무 적용 방안

### 가. 실무 도입 시 고려사항: 인터페이스 우선 원칙
- **Interface over Abstract Class**: 다중 상속이 불가능한 추상 클래스보다, 유연한 설계가 가능한 인터페이스를 우선적으로 사용하여 결합도를 낮추어야 함
- **추상 클래스의 보완**: 공통 로직이 많고 상태(Fields) 공유가 필수적인 경우에만 제한적으로 추상 클래스 도입

### 나. 거버넌스 및 보안(Security) 통제 방안
- **API 거버넌스**: 외부 모듈과의 통신 시 반드시 인터페이스를 통해 소통하게 함으로써 내부 구현 정보를 보호(**Information Hiding**)
- **DIP(의존성 역전) 적용**: 고수준 정책이 구체 클래스가 아닌 인터페이스(추상화)에 의존하게 하여 보안 패치나 모듈 교체 시 리스크 최소화

### 다. 현대적 트렌드와의 연계
- **Microservices (MSA)**: 서비스 간의 통신 규약(IDL, Swagger)은 일종의 **거대 인터페이스** 역할을 수행하며, 이를 통한 계약 기반 설계(Design by Contract)가 핵심
- **Cloud Native**: 인프라 환경에 구애받지 않는 이식성(Portability) 확보를 위해 인프라 제어 로직을 인터페이스로 추상화하는 전략 필수

> [!tip] **기술사 인사이트**
> 추상 클래스는 **"부모의 재산을 물려받는 것"**이고, 인터페이스는 **"자격증을 따는 것"**과 같습니다. 기술사 답안에서는 이 둘의 문법적 차이를 넘어, **시스템의 결합도를 제어**하고 **변경의 파급효과를 차단**하기 위한 전략적 선택의 근거를 명확히 제시해야 합니다.
